    Synthetic vs. real logs — why this view looks so clean read me

    This demo feeds the model synthetic logs: one line per event, one unified format, line IDs, and the root cause spelled out in a tidy line. Real deployments are two separate, far noisier streams where the cause is not announced — you have to infer it.

    This demo · synthetic
    [L005] 10:42:00.458 WARN  AMF  [amf] WARNING: Registration reject (cause=Illegal UE, auth failure)
    • one line per event, one format
    • line IDs (L005) for evidence
    • root cause stated outright
    Real · Open5GS (5G core)
    10/15 09:24:02.001: [amf] WARNING: [imsi-999…] Authentication failure(MAC failure) (../src/amf/gmm-sm.c:1204)
    • date + µs, [nf] tag, src file:line
    • mostly INFO — failures are buried
    Real · srsRAN (gNB / RAN)
    2023-12-15T09:24:01.34 [NGAP  ] [I] TX InitialUEMessage
[NGAP  ] [D] Rx PDU (28 B): 00 15 40 18 …
    • ISO ts, [LAYER] tags, hex/ASN.1 dumps
    • a separate stream from the core

    So a strong score here means "reads clean synthetic logs well" — not yet "works on messy real logs." Capturing real testbed logs is the honest next step.
